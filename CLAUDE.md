## Деплой
Проект на GitHub: github.com/taiwoismad-ui/my-project

### Структура веток
- `gh-pages` — боевая ветка, именно отсюда GitHub Pages раздаёт сайт
- `claude/*` — рабочие ветки сессий Claude (создаются автоматически)

### Файл планировщика
- Живёт в: `gh-pages/planner/index.html`
- Статические ресурсы (иконки и т.п.): `gh-pages/planner/assets/`
- Манифест PWA: `gh-pages/planner/manifest.json`

### Как деплоить изменения
1. Внести правки в `index.html` на рабочей ветке
2. Переключиться на `gh-pages`: `git checkout gh-pages`
3. Скопировать файл из рабочей ветки:
   `git show <рабочая-ветка>:index.html > planner/index.html`
4. Закоммитить и запушить:
   `git add planner/index.html && git commit -m "..." && git push origin gh-pages`
5. Вернуться на рабочую ветку: `git checkout <рабочая-ветка>`

Сайт обновляется через 1–2 минуты после пуша в `gh-pages`.

## Типовые сценарии

### Если задача большая (3+ экрана или функции)
- Разбей на подзадачи
- Покажи план списком
- Жди ок перед стартом
- Выполняй по одной подзадаче за раз

### Если что-то сломалось после твоих изменений
- Не продолжай следующую задачу
- Сначала найди причину бага
- Предложи откат или точечный фикс
- Зафиксируй ошибку в разделе "Известные грабли"
- Жди решения от меня

### Если задача требует внешних сервисов (API, OAuth, сервер)
- Честно скажи что это выходит за рамки single-file подхода
- Предложи упрощённый вариант который работает без сервера
- Объясни плюсы и минусы каждого варианта

### Если непонятно что именно нужно сделать
- Не додумывай
- Задай один конкретный вопрос
- Не несколько вопросов сразу — только один самый важный

## Запреты
- Не подключай npm-пакеты без явного разрешения — проект без сборщика
- Не меняй логику Firebase без явного запроса — только UI и CSS
- Не удаляй существующий функционал при добавлении нового
- Не пуши в main без явной команды "задеплой"
- Не используй localStorage для данных приложения — только Firebase

## Приоритеты при конфликте задач
1. Не сломать то что уже работает
2. Мобильная версия важнее десктопа
3. Простое решение лучше умного но сложного
4. Скорость итерации важнее идеального кода

## Самообновление при ошибках
Если ты допустил ошибку которая сломала существующий функционал:
1. Исправь ошибку
2. Добавь в раздел "Известные грабли" описание ошибки и правило которое предотвратит её повторение
3. Сообщи мне что добавил и что именно написал

Формат записи в "Известные грабли":
- [Что сломалось] → [Почему] → [Правило на будущее]

Пример:
- Вкладки вишлиста сломались после правки заметок → общий CSS-класс .tabs перезаписался → всегда используй уникальные классы для каждого раздела: .wishlist-tabs, .notes-tabs

## После каждой задачи
1. Скажи что именно сделано (списком)
2. Напомни команду для деплоя если нужен git push
3. Предложи следующий логичный шаг
4. Если за сессию выполнено 2+ задачи — предложи обновить CLAUDE.md и дай готовый текст для вставки в разделы "Текущее состояние" и любые другие которые изменились
